- [x] T001 Extend `client/src/test-harness/RecordingDriftHarness.svelte` (already merged, with `hostBacking` / `participantSynth` / correction-off modes) to drive a backing-track **host** against a backing-track **participant**. This backing/backing direction has never been exercised — the prior run only measured synth-host directions plus a partial backing-host case — and it is the exact configuration this plan depends on. The harness must expose, per sample: each side's `tickPosition`, each side's own `HTMLAudioElement.currentTime`, and the resulting separation in both ticks and milliseconds. Milliseconds is the reporting unit that matters; ticks are tempo-relative and mislead across tempos. No test requirement — this is harness capability that T002 immediately exercises.

- [x] T003 Implement backing-host rate-keying in `client/src/playback-sync.ts` so the drift projection uses the recording's sync-point-derived rate when the host is on a backing track, rather than `localTempoAtTick`'s notated tempo (feedback F003(ii)). alphaTab keeps sync points in `MasterBar.syncPoints`, a separate collection from `MasterBar.tempoAutomations`, so a tempo-automation walk can never observe the effective rate — this is confirmed, and drift accumulates at exactly `Δbpm × 16` ticks/sec. **Do not alter `correctDrift`'s existing arithmetic**: it was instrumented and is exact (host deviation from its own projection is 0 ticks, flat). This task supplies a correct *rate input*, not a correction term. Extend `client/src/playback-sync.test.ts` with unit coverage of the rate derivation before implementing. [artifacts: infrastructure]

- [x] T004 Solve the per-`play()` start skew for the uniform-recording case, turning T002 green. Constraints, all established by measurement (`research-recording-mode-drift-2026-07-19-b7c2.md`, T004a/T004b): the skew is ~275ms, invariant under `bufferTimeInMilliseconds`, stable to ±3ms within a playback, and **re-rolled on every start** (275→342ms across a seek) because `HTMLAudioElement.play()` completes asynchronously after decode/buffer — so **no compensation constant can exist**. `correctDrift`'s arithmetic must not be touched. Any mechanism that re-measures skew *after* the participant has drifted will ratchet, re-absorbing drift as skew (this is exactly how the prior attempt failed). A viable approach must therefore derive the skew from a reference that cannot be contaminated by the correction it feeds — note both clients' positions track their own `audio.currentTime` to ±5ms, which is the one trustworthy local signal available. **If no approach meets T002's assertion, STOP and report rather than relaxing the assertion.** [artifacts: infrastructure]

- [x] T005 Extend T002's spec with the same ≤50ms end-state separation assertion on the `recording-skewed` fixture (Δbpm=10), backing/backing. Both clients hear identical audio, so tempo divergence from the notated score must not separate them — a failure here means T003's rate-keying is incomplete, not that the bar is wrong. This is the case that isolates the projection-rate problem from the start-skew problem.
