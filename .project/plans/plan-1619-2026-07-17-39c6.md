# Plan: Lyrics-timing fixes, mute-parts tab, and playback-start stutter

## Goal

Fix the confirmed `.lrc` line-timing drift and the unsound melisma-dedup
heuristic on GP files without author line breaks, investigate the in-tab
ticker's reported offset as a possible audio-output-latency issue, move the
per-part mute control into its own Settings tab, and investigate/fix
playback's start-of-song audio stutter.

## Scope

**In scope:**
- `feedback-lyrics-timing-tiro-c741.md` F002 (`.lrc` line-boundary drift)
  and F003 (unsound melisma dedup + pipeline/client walk divergence) — both
  confirmed root causes with a concrete fix direction.
- `feedback-lyrics-timing-tiro-c741.md` F001 (ticker ~2-syllable-ahead) —
  **investigate only** this pass, per explicit user choice: the user
  confirmed they were on Bluetooth/wireless audio output when they noticed
  it and haven't compared against wired/built-in speakers. The leading
  hypothesis (uncompensated `AudioContext` output latency) could not be
  reproduced as a code-level bug against local low-latency output, so a
  fix is not committed to in this plan — only a research task to confirm
  or rule out the hypothesis, with an implementation task gated on its
  outcome.
- `feedback-mute-parts-own-tab-cf6d.md` F001 — promote "Mute parts" out of
  the Preferences tab into its own dedicated Settings tab, one row per
  part.
- `feedback-playback-start-stutter-2052.md` F001 — investigate what
  alphaTab's audio engine exposes for pre-buffering/warm-up, implement a
  fix if a clean mechanism exists.

**Out of scope:**
- The `solo-mute-button-per-part` backlog feature (a separate, already-filed
  new capability — not this plan's concern; plan it separately via
  `/ardd-plan solo-mute-button-per-part`).
- Any change to the pipeline's *primary* (GP-has-line-breaks) extraction
  path — only the lrclib-fallback branch (`extractLyrics`'s `else` at
  `packages/pipeline/src/extract-lyrics.ts:58-66`) is affected by F002.
- Cursor-position latency compensation (the tab cursor is presumably
  equally affected by any confirmed output-latency issue, per F001's
  research finding — its continuous animation just masks the same lead)
  is not committed to in this plan; if F001's research task confirms the
  hypothesis and a fix is implemented, extending the same compensation to
  the cursor is an explicit Open Question below, not committed scope.

## Technical Approach

### F002 + F003: shared, timestamp-grounded syllable walk (Phases 1-2)

Both bugs trace back to the same root: pipeline (`packages/pipeline/src/
gp-parser.ts#extractSyllables`) and client (`client/src/
lyrics-beat-walk.ts#walkLyricBeats`) independently re-implement "walk a
GP track's beats into a flat syllable stream," and they've drifted apart:

| | Pipeline (`gp-parser.ts`) | Client (`lyrics-beat-walk.ts`) |
|---|---|---|
| Tick source | `bar.masterBar.start + beat.playbackStart` (pre-T004 formula — reopens the grace-note bug T004 fixed client-side, though a no-op on this specific file) | `beat.absolutePlaybackStart` (correct, post-T004) |
| Dedup | none | same-text ⇒ collapse (wrongly collapses genuinely-repeated distinct syllables like "yeah, yeah, yeah") |

**Fix: extract one shared implementation into `packages/shared`** (both
`client` and `packages/pipeline` already depend on `@coderline/alphatab`
directly at the same version range, and the walk itself is plain
TypeScript with no browser-only API — it runs identically in Node).
`packages/shared/src/lyrics-walk.ts` exports `walkSyllables(score, source):
Syllable[]`, using `beat.absolutePlaybackStart` for every tick and a
**tie-aware** dedup: collapse a beat into the previous syllable only when
its text matches **and** at least one of its notes is a tie continuation
(`beat.notes.some(n => n.isTieDestination)`) — never on text match alone.
This is the actual discriminator between "one sustained syllable spread
across tied beats" (should collapse) and "the same word sung twice as
distinct notes" (should not). Both `lyrics-beat-walk.ts` and
`gp-parser.ts#extractSyllables` become thin call-throughs to this shared
function, so they can never diverge again (constitution Principle II).

**F002's actual fix**, once the shared walk exists: replace
`extract-lyrics.ts`'s `distributeByWordCount()` (word-count-proportional
*rounding* — confirmed wrong: 41 of 44 lines mis-assigned on the reported
song) with **timestamp-based line-boundary placement**. lrclib's synced-
lyrics blob already carries a real `mm:ss.xx` timestamp per line —
`lrclib.ts#parseLrclibLines` currently **discards** it
(`packages/pipeline/src/lrclib.ts:25-30` strips the `[mm:ss.xx]` prefix
before returning). Parse it instead, convert each line's timestamp to a
tick via an inverse of `gp-parser.ts#buildTickToMs`'s tempo-segment walk
(add a sibling `buildMsToTick`, sharing the same tempo-segment builder —
Principle II again, not a second duplicated tempo-map walk), and for each
line find the syllable-stream index whose tick is closest to that line's
tick. The gap between consecutive line-boundary indices becomes that
line's `lyricLineBreaks[i]` — grounded in the syllable stream's own
already-correct GP ticks, immune to the textual mismatches between GP and
lrclib wording that broke word-count matching (e.g. this song's GP lyrics
read "You will be the death of me", lrclib's read "And you will be the
death of me" — one extra word `distributeByWordCount` had no way to
account for).

### F001: latency-compensation research (Phase 3)

Not reproducible as a code bug against the dev machine's local audio
output (ticker/engine-tick/audible-onset all agreed within ~25-40ms
across dozens of syllables, live-instrumented via an `AnalyserNode`). The
user confirmed Bluetooth output, untested against speakers — a 300-500ms
Bluetooth output latency at this song's 118bpm is ≈960 ticks, matching
the reported "2 syllables ahead" almost exactly. Research task: determine
whether alphaTab's player exposes (or can be made to expose) the
underlying `AudioContext` so `outputLatency`/`baseLatency` can be read and
converted to ticks for a compare-tick offset in `lyrics-overlay.ts`'s
highlight logic. alphaTab's own type declarations don't surface these
directly (checked `dist/alphaTab.d.ts`) — reaching them may require
digging into the player/synth's internals, which is why this is a
research task before an implementation task, not a committed fix.

### Mute-parts own tab (Phase 4)

`client/src/components/SettingsModal.svelte` currently has three tabs
(`'participants' | 'session' | 'preferences'`, line 19) and renders "Mute
parts" as a single wrapped `.control-row` of buttons inside the
Preferences branch (lines 219-231) — all parts share one flex-wrapped row,
not one row each. Add a fourth tab (`'tracks'` or similar), move the
existing `trackMutes`/`toggleTrackMute` logic and markup there unchanged
in behavior, and restructure the markup so each part renders on its own
row (a `<div class="control-row">` per part, not one shared row for all
parts) — matching the feedback's explicit "1 line per part" ask.

### Playback-start stutter (Phase 5)

`client/src/readiness.ts#waitUntilReady` resolves the moment
`api.scoreLoaded` and `api.soundFontLoaded` both fire — there's no
additional audio-engine warm-up step between "ready" and the first note
actually playing. This is consistent with a classic Web Audio first-buffer
glitch: an `AudioContext` created/resumed just before playback starts
(likely on the user gesture that presses "Start") can stutter on its first
few callbacks while the audio graph and any worklets spin up. Research
task: identify what, if anything, alphaTab's player exposes for a silent
pre-roll or explicit `AudioContext.resume()` ahead of the actual `api.
play()` call, and whether triggering that earlier (e.g. at Lobby-load time,
well before "Start" is pressed) eliminates the stutter. Implement if a
clean mechanism exists; if not, report back with findings as an
alternative feedback item rather than forcing a hacky fix.

## Open Questions

- **F001**: if T014 confirms the output-latency hypothesis, should the
  same compensation extend to the tab cursor (equally affected, per the
  research finding, just masked by its continuous animation)? Explicitly
  out of this plan's committed scope (see Scope) — worth a follow-up
  feedback item if confirmed, rather than silently expanding this plan's
  surface.
- **F001**: is there a reliable way to read `AudioContext.outputLatency`
  across alphaTab's supported browser/environment matrix, or only in some
  (e.g. Chrome-only support for `outputLatency` historically lagged
  `baseLatency`)? T014 needs to establish this before T015 can commit to
  an approach.
- **Phase 5**: is the stutter audio-only, cursor-only, or both? The
  original feedback said "audio and/or cursor" — T019 should establish
  which, since a cursor stutter (a rendering/animation issue) and an audio
  stutter (a Web Audio buffering issue) would need different fixes.
