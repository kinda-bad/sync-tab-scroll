# Plan: Fix Spotlight mode force-follow

## Goal

Make Spotlight mode's force-follow actually move a participant's view when the host sets the lobby cursor while Spotlight is on, and lock the fix in with a CT spec so it can never silently regress again.

## Scope

**In scope:**
- Diagnosing why `client/src/playback-engine.ts`'s Spotlight-follow block (the `clientStore.subscribe` callback that applies `api.tickPosition = session.lobbyCursorTick` when `spotlightMode` is true) never visibly applies, even once the alphaTab engine is fully warmed up (confirmed via live real-playback testing during T010 manual verification).
- A new Playwright CT spec that reproduces the failure deterministically (red first, per constitution Principle VII), using the existing `__getApi`/`isReadyForPlayback`-polling pattern already established in `full-lyrics-view.ct.spec.ts`.
- The actual fix in `playback-engine.ts` (and nowhere else, unless the root cause turns out to live elsewhere â€” see Open Questions).
- Closing out `tasks-lobby-cursor-modes-0bea.md` T010: re-verifying scenario 2 live in two real browser tabs once the fix lands, and checking the task off with that evidence.

**Out of scope:**
- Scenarios 1 (Spotlight off â†’ no follow) and 3 (auto-reset on Start) â€” both already verified passing live; this plan does not re-litigate them beyond a regression pass.
- Any broader refactor of `isReadyForPlayback` gating elsewhere in `playback-engine.ts` (e.g. the click-to-seek gate) unless the root cause is that shared gate and fixing it there is the only correct fix.
- `ui.md`'s Spotlight description â€” it already correctly describes the *intended* behavior; this is a code bug, not a documentation gap, so no artifact content changes are expected. (Only `last_updated`/`diagram_status` bookkeeping, if any artifact happens to get touched incidentally.)

## Technical Approach

`ui.md` and `datamodel.md` already fully specify Spotlight mode's intended behavior (`spotlightMode: boolean` on `Session`, force-follow gated on it, reset to `false` on playback start) â€” this plan does not change that design, only the implementation that currently fails to deliver it.

Per constitution Principle VII (Test-First Development), the fix is driven by a failing test first: a new CT spec exercises `playback-engine.ts`'s Spotlight-follow subscription directly against a mounted component and a real (or faked-just-enough) `clientStore`, sets `spotlightMode: true` and a `lobbyCursorTick`, and asserts `api.tickPosition` (or the visible cursor/scroll effect) actually changes. This spec is written and confirmed red against the current code before any fix is attempted â€” it is expected to reproduce exactly what live manual testing already found broken.

Root-cause diagnosis starts from the prime suspect identified during live verification: `playback-engine.ts`'s `clientStore.subscribe` callback (around lines 475-517) early-returns via `if (!s.session || !api.isReadyForPlayback) return;` before ever reaching the Spotlight-follow block. Live testing ruled out simple timing (the block still didn't fire even after a full real Startâ†’playbackâ†’Stop cycle proved the engine was fully functional and `isReadyForPlayback`-dependent behavior â€” real playback cursor movement â€” worked correctly), so the actual defect is either: this gate genuinely staying false for the lobby/pre-playback participant instance in a way not yet understood, the `clientStore.subscribe` callback not re-firing on every `session-state` broadcast the way assumed, or a bug local to the Spotlight-follow block itself (e.g. the `lastAppliedSpotlightTick` guard never resetting appropriately, or a stale closure). The CT spec is designed to make this observable and debuggable in isolation, rather than diagnosed further via live two-tab testing.
