# Plan: Lyrics ticker tie-boundary early-highlight fix

## Goal

Root-cause and fix the in-tab lyrics ticker highlighting a syllable too early across a tie/barline boundary (the "TIRO" measure 8→9 repro), and close out the still-open audio-output-latency research thread as superseded by this sharper, non-latency repro.

## Scope

**In:**
- Diagnose why the ticker highlights "You're" (the syllable belonging to the first note of measure 9) while playback is still on measure 8's last (tied, breath, no-lyric) note.
- Instrument/reproduce the tie-across-barline case as a unit test against `packages/shared/src/lyrics-walk.ts` (the shared syllable walk) and, if the root cause turns out to live in the client's tick-compare logic instead, against `client/src/lyrics-overlay.ts`'s `updateActiveSyllable`.
- Fix the root cause once diagnosed, with a regression test that encodes the measure-8/9 tie scenario so it can't silently regress.
- Close out `feedback-audio-output-latency-t014-dfa8.md` F001 formally: document that this plan's fix addresses the reported symptom via a different (non-latency) root cause, so the Bluetooth-output-latency hypothesis is deprioritized rather than pursued further this pass.

**Out:**
- Implementing audio-output-latency compensation using alphaTab's unsupported `output.context` internal (per that feedback's own recommended-next-step, this still needs real multi-device browser access to validate and is not blocking — if the "syllable(s) ahead" symptom persists after this plan's fix ships, that's grounds for fresh feedback, not scope creep here).
- Any other open feedback file — this plan is scoped to exactly the two feedback items above (`/ardd-plan` feedback-scope).

## Technical Approach

`client/src/lyrics-overlay.ts`'s `updateActiveSyllable(tickPosition)` walks the flattened syllable list and selects the *last* syllable whose `tickPosition <= tickPosition` (line ~145) — that comparison is correct by construction *given* accurate `tickPosition` values. `packages/shared/src/lyrics-walk.ts#walkSyllables` is the sole source of those `tickPosition`s (`beat.absolutePlaybackStart`) and owns the tie-continuation dedup logic (`isTieDestination`).

Two candidate root causes, to be distinguished empirically rather than guessed:

1. **Walk/dedup misattribution** — the tie-continuation check (`beat.notes.some(n => n.isTieDestination)`) may not correctly handle a tie that crosses a barline into a beat with *no* lyric text of its own (measure 8's last note: a tie destination, breath, no syllable) followed immediately by a beat that *does* carry the next lyric ("You're", measure 9 first note). If the walk's per-beat iteration or tie logic causes "You're"'s syllable entry to inherit an earlier `tickPosition` than `beat.absolutePlaybackStart` actually reports for measure 9's first beat, the highlight would jump early exactly as described.
2. **Client tick-compare / lookahead issue** — a bug in how `client/src/lyrics-overlay.ts` or `client/src/lyrics-beat-walk.ts` compares the live playback tick against the flattened syllable array (e.g. stale index caching, an off-by-one in the `<=` boundary, or a lookahead added by a prior lyrics-timing pass) could show the same symptom even with correct `tickPosition`s.

Diagnose first with a focused unit test against `walkSyllables` using a constructed tie-across-barline fixture (mirroring the existing tie tests in `packages/shared/src/lyrics-walk.test.ts`) to establish whether the *tick values themselves* are wrong. If they're correct, move to `client/src/lyrics-overlay.ts`'s comparison logic. This mirrors the diagnose-then-fix structure `plan-1619-2026-07-17-39c6.md` already used successfully (its T014/T015 split) — a plan that also touched this exact code path.
