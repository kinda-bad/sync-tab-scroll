# Plan: Lyrics overlay timing + display fixes

## Goal

Fix the lyrics-overlay syllable-highlight timing offset and five related
display bugs in the lobby/lyrics/count-in/gap UI, per
`feedback-lyrics-overlay-timing-display-5456.md`.

## Scope

In scope: the six items from the feedback file (F001тАУF006), all confined to
`client/src/lyrics-overlay.ts`, `client/src/playback-engine.ts`,
`client/src/views/Lobby.svelte`, `client/src/App.svelte`, and
`client/src/lyrics.css`.

Out of scope: any other playback-sync behavior not named in the feedback
file; no artifact changes are anticipated (see Technical Approach) since
these are implementation bugs, not decision reversals.

## Technical Approach

Investigation (read-only) found:

- **F001 (timing offset)** тАФ `updateActiveSyllable(tickPosition)` in
  `lyrics-overlay.ts` (~L71-86) selects the last syllable with
  `tickPosition <= tickPosition`. The reported "~2 syllables ahead" symptom
  points at the syllable data's `tickPosition` values themselves running
  ahead of the audio's true tick (or a stale/pre-buffered position feed),
  not the selection logic тАФ needs an implementation task to instrument and
  compare `api.playerPositionChanged`'s tick against actual audio time and
  correct the offset at its source (in the syllable-to-tick mapping, or in
  the event's reported tick, whichever is found to drift).
- **F002 (duplicate scrollbars / lobby never scrolls away)** тАФ root cause
  confirmed: in `App.svelte` (~L128-142), `.app-content` (wrapping
  `<Lobby/>`) and `.full-lyrics-view` are top-level siblings, toggled
  independently (`hasPart && isLyricsPart`), not nested in one scroll
  container. `.app-content` has no height/overflow constraint, so it just
  sits in normal document flow above the lyrics sheet, which manages its
  own internal scroll (`overflow-y: auto` in `lyrics.css` `.full-lyrics-view`,
  ~L31-47). Fix: unify into a single scroll region, or explicitly
  hide/collapse `.app-content` when the lyrics view is active so only one
  scrollable region exists.
- **F003 (drain bar doesn't disappear)** тАФ `renderGapIndicators()` /
  `updateGapIndicators()` in `playback-engine.ts` (~L178-228) never removes
  or hides the `.gap-drain` element once `currentTimeMs` passes
  `gap.endMs` тАФ fill clamps to 0 but the element stays in the DOM. Fix:
  remove/hide the gap indicator element once its gap has fully elapsed.
- **F004 (count-in dots don't disappear)** тАФ same `renderGapIndicators()` /
  `updateGapIndicators()` path, same missing hide-after-gap-ends logic as
  F003 (the dots are children of the same gap-indicator element). Fix
  alongside F003 in the same indicator-lifecycle change.
- **F005 (first line pre-highlighted)** тАФ two contributing sites:
  `playback-engine.ts`'s `.lrc` fetch handler calls `setActiveLine(0)`
  (~L248) immediately on load, before playback starts, and its
  `playerPositionChanged` handler (~L258-271) defaults `index = 0` rather
  than `-1`. Also `lyrics-overlay.ts`'s `<=` comparison in
  `updateActiveSyllable` (~L74) can flip the first syllable active at tick
  0 if `flat[0].tickPosition === 0`. Fix: don't mark line/syllable 0 active
  until real playback position data confirms it's reached (index starts at
  -1 / "none active" until the first genuine position update clears it).
- **F006 (count-in dots redesign: inline prefix)** тАФ currently
  `.gap-indicator` (`lyrics.css` ~L74-80) is `flex-wrap` with `.gap-drain`
  forced onto its own line (`flex-basis:100%`), and the whole indicator is
  `insertBefore`'d ahead of the `.lyric-line` in `playback-engine.ts`
  (~L206) тАФ i.e. dots render above the upcoming line. Redesign: render the
  count-in dots as an inline prefix to the upcoming lyric line's own text
  content (e.g. "┬╖┬╖┬╖┬╖ You will be"), removing the separate
  above-the-line placement.

No artifact changes are anticipated тАФ these are implementation-only fixes
to existing UI/sync mechanisms already described in `ui.md`; nothing here
reverses a documented decision.

## Phase Breakdown

### Phase 1: Gap-indicator lifecycle (F003, F004)
- [ ] T001 [artifacts: ui] Add hide/remove-after-gap-ends logic to
  `renderGapIndicators`/`updateGapIndicators` in `playback-engine.ts` so
  the drain bar and count-in dots clear once `currentTimeMs` passes
  `gap.endMs`. Test-first: a playback-engine unit/CT test asserting the
  gap-indicator DOM node is absent/hidden once past `gap.endMs`.
  [feedback: F003, F004]

### Phase 2: Count-in dots as inline lyric prefix (F006)
- [ ] T002 [artifacts: ui] Redesign gap-indicator rendering so count-in
  dots render as an inline prefix on the upcoming `.lyric-line`'s text
  (e.g. "┬╖┬╖┬╖┬╖ You will be") instead of a separate `insertBefore`
  element above the line; remove the now-unused above-line CSS
  (`flex-basis:100%` drain-bar-on-own-line rule) as needed. Depends on
  T001's lifecycle change existing so the prefix state resolves cleanly
  once the count-in ends. Test-first: a CT test asserting the dot prefix
  renders attached to the lyric line and clears/resolves into the plain
  line text at the right time. [feedback: F006]

### Phase 3: First-line/syllable pre-highlight fix (F005)
- [ ] T003 [artifacts: ui] [parallel] In `playback-engine.ts`, stop calling
  `setActiveLine(0)` on `.lrc` load and change the `playerPositionChanged`
  default from `index = 0` to `index = -1` (no line active) until a real
  position update confirms line 0's timestamp has been reached. In
  `lyrics-overlay.ts`'s `updateActiveSyllable`, guard the first syllable
  so `tickPosition === 0` doesn't flip active before a genuine position
  event arrives. Test-first: a test asserting no line/syllable is `.active`
  immediately after load, only after playback reaches the first
  timestamp. [feedback: F005]

### Phase 4: Syllable highlight timing offset (F001)
- [ ] T004 [artifacts: ui] [parallel] Instrument
  `api.playerPositionChanged`'s reported tick against actual audio
  playback time for a known song (TIRO or Creep) to locate where the
  ~2-syllable-ahead offset originates (syllable `tickPosition` data vs.
  the position event itself), then correct it at the source. Test-first:
  a regression test pinning a specific tick тЖТ expected-active-syllable-index
  mapping for a sample syllable list, covering the previously-off-by-2
  case. [feedback: F001]

### Phase 5: Lobby/lyrics single scroll region (F002)
- [ ] T005 [artifacts: ui] In `App.svelte`, unify `.app-content` (wrapping
  `<Lobby/>`) and `.full-lyrics-view` into a single scroll container (or
  explicitly hide/collapse `.app-content` when the lyrics view is active)
  so only one vertical scrollbar exists and the lobby content can fully
  scroll out of view. Test-first: a CT test asserting only one scrollable
  container is present when the lyrics view is active, and that lobby
  content is not visible/interactive once scrolled past.
  [feedback: F002]

## Open Questions

- F001's true root cause (syllable data vs. position-event tick) is not yet
  isolated тАФ T004 starts with instrumentation/measurement, not a known
  fix, so its scope may grow once the actual offset source is found.
- F006's exact prefix rendering approach (styling of the dots as inline
  text vs. a small inline component) is left to implementation judgment
  during T002.
