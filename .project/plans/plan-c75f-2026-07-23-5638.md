# Plan: Nav/host-control feature bundle + open feedback + surfaced defects

## Goal

Deliver four backlogged nav/host-control features (help/about panel,
host-mandated bars-per-row layout, host early-stop point, remembered
display name), close out four open feedback items (e2e-suite-red,
input-sanitization hardening, join-code click-to-copy, and the
recording-mode metronome-lock reconsideration), and fix the two
currently-unsurfaced `DEFECTS.md` entries (alphaTab worker script URL
comment drift, Tracks tab "Mute all" control).

## Scope

**In scope:**
- Help/Info/About nav panel (static content, no server round-trip)
- Host-mandated bars-per-row layout pin + personal per-participant density preference
- Host early-stop-point control (tick-based auto-stop + de-emphasis)
- Remembered display name pre-fill for signed-in users
- Join-code click-to-copy
- Server-side displayName/activation-key input validation
- Diagnosing and resolving (or explicitly documenting) the red e2e `host-controls.spec.ts` suite
- A "green means what tiers ran" reporting convention (constitution)
- Recording-mode metronome preference no longer force-disabled (visual beat widget stays interactive; audible layering is an open question to resolve during implementation)
- Fixing DEFECTS.md bf07f912 (stale worker-script-URL comment) and b16e2ab1 (Mute-all control drift)

**Out of scope:**
- Any other backlogged feature not listed above
- Any other open feedback file not listed above
- Rate limiting, in-app authoring Phase 2 work, or any other constitution-flagged future direction not touched by these items

## Technical Approach

Nav/host-control features follow existing session-setting patterns
already established for `countInEnabled`/`spotlightMode`/`lobbyCursorTick`
(infrastructure.md Session & Real-Time Sync): a host-authorization-checked
WS message sets a `Session` field, the ordinary `session-state` broadcast
carries it to everyone, no bespoke result message. `Session.hostBarsPerRow`
and `Session.earlyStopTick` (datamodel.md) are added as two more fields in
this family. The early-stop enforcement reuses the existing host-Stop
transition path (`playbackState.status` flip), triggered server-side once
the host-reported `tickPosition` passes the threshold, rather than a new
mechanism. Bars-per-row applies at alphaTab renderer construction
(infrastructure.md Tab Rendering), pinned when set and falling back to
each participant's own client-local preference otherwise — same
override-shadows-preference pattern the design already uses for e.g. the
metronome vs. recording-mode carve-out.

Remembered display name and join-code click-to-copy are client-only
changes (Landing View / Bar identity area, ui.md) with no new wire
messages. Input validation is a single shared server-side function
applied at the point `session-create`/`session-join`/`catalogue-unlock`
are handled (infrastructure.md), not a client-side change.

The e2e-red investigation (F001) is diagnostic-first: establish whether
`host-controls.spec.ts`'s timeout-shaped failure is environmental
(harness/config) or a real regression before attempting a fix, per the
feedback's own suggested first step (bisect against the 2026-07-19 known-
good date). The reporting-convention feedback (F002) is a one-line
Quality Standards addition to `constitution.md` (already applied, v1.6.2).

The metronome reconsideration (feedback-recording-mode-metronome-lock-
reconsidered-c415 F001) keeps the toggle interactive during recording
playback for the beat widget's visual component; whether an audible click
can be layered over the recording is explicitly left as an implementation-
time investigation against alphaTab's upstream #1961 constraint — ui.md
already documents both possible outcomes.
